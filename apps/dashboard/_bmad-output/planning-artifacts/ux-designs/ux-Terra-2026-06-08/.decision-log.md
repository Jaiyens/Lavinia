## North star (the reframe that governs everything)
The home screen's job is to make the grower **feel they can see their whole farm and know what is happening on it** — situational awareness and a feeling of control, not "here is a number." The dashboard is the product's pitch; it should sell itself the moment real data is in.

## Product framing
- This repo is **Tool 1: the PG&E energy tool**, but it is built as the first **agent/tab inside a farm operating system**. Design the shell now so Tool 2+ slot in mechanically.
- Build for anyone who signs up, with Batth (57 accounts / ~183 meters) as the scale stress-test, not the template.
- Honest lever priority unchanged: rate optimization first; staggering stays demoted. Planner, not live meter (PG&E data lags ~1 day; never promise real-time).

## Information architecture — the three-zone shell ("inverted-L + copilot")
- **Left — agent rail (the OS shell).** Lists *agents*, not features. v1: Home + Energy (live). Future agents (Water, Labor, ...) render visible-but-disabled ("coming") to sell the OS without faking capability. Desktop = left rail; mobile = bottom tab bar.
- **Center — the data hero.** A **lens toggle over ONE meter dataset**: **Chart · Table · Map · Calendar**, with the **TOU-stacked cost Chart as the default face**. One lens visible at a time (the discipline that prevents the 12-widget wall). A compact KPI strip sits above the lens (small number + sparkline + delta each — never a lone hero number).
- **Right — persistent findings rail.** The agent's findings beside the data on every agent screen: money hiding, rate fixes, propose→approve actions. Money is surfaced here, not as the center hero.
- **Shared drawer.** Clicking a chart bar / table row / map pin opens one meter-detail drawer (same drawer from every lens).
- **Home = the Energy dashboard today**, and grows a thin cross-agent roll-up strip above the hero the day a second agent ships. Data model already supports this via `Recommendation.tool`.

## Hero decision (overrides PRD "money is the largest element")
- Growers read **graphs + tables**, not a lone dollar figure (validated by the Wexus teardown: Wexus leads its dashboard with TOU-stacked energy/cost charts; the research's card→chart→table→drawer pattern never prescribes a giant number).
- Therefore: **hero = chart + table**; money is the *story those visuals tell*. The KPI strip carries compact numbers (spend / demand exposure / biggest mover), each paired with a sparkline + delta.

## The map (promoted from PRD P1 to a co-equal home lens)
- The map is the single strongest "feel of control" element. Pins are colored by **$ at risk / attention** (traffic-light semantics), not vanity health.
- **Geometry comes from the bill, no Bayou needed.** PG&E bills do NOT print lat/long. They locate each meter two ways: (1) **PLSS legal descriptions** in the "Service For" line — e.g. "NW NE 33-16-19" = NW¼ of NE¼ of Section 33, Township 16S, Range 19E, Mount Diablo Meridian (~40-acre cell); the Batth account spans T15–17S / R18–20E around Caruthers. (2) **street addresses** for shops/offices. Convert PLSS→centroid via BLM PLSS/CadNSDI; geocode addresses directly. Master spreadsheet (lat/long + GPM) is the cleaner source when it lands.
- Map is zoomable; "looks good and cool" is an explicit goal, balanced by the comprehension bar.

## Findings rail placement (chosen by Jaiyen)
- **Persistent right rail** on every agent screen. On mobile it collapses to a **peeking bottom sheet** (e.g. "3 findings · ~$78k ↑"). `[ASSUMPTION]` bottom-sheet is the mobile collapse — Jaiyen approved right-rail; mobile form is my call.
- v1 is **display-only**: the "approve" affordance is shaped for the agent to execute later but only displays/records in v1 (Recommendation grammar).

## Onboarding (value-honest, operator-operable — NOT a scripted reveal)
Jaiyen rejected the theatrical "one bill → reveal → add the rest" drip. Real context: the grower is usually already sold (post-demo) or Jaiyen is on-site setting it up for them. The dashboard is the pitch; onboarding just gets data in and gets out of the way.

Flow: **Identify → Connect a data source → land in the real dashboard.**
- **Identify:** name + contact. Operable by Jaiyen on behalf of the farmer.
- **Connect your data — pick how, choose how many accounts:**
  - Connect PG&E (Share My Data authorization / LOA) → pulls the authorized accounts, stays current.
  - Upload bills (one per account) → we read meters, rates, cost, and location off each.
  - Upload meter-master spreadsheet → bulk inventory across many accounts (optional accelerator).
  - **Gate = one real source** (PG&E authorization OR billing). No PG&E? fine, skip. No spreadsheet? fine, optional. Account scope is the user's choice; add more anytime.
- **Tour a sample** door into the badged representative (demo) dashboard for the curious — zero commitment.
- A lightweight **confirm** step handles "we couldn't read this from the bill."
- In v1, **bill upload is the workhorse**; the live connection (PG&E authorization / Bayou) is the "so you never upload again" upgrade (Bayou parked in v1).
- Reference: Wexus onboarding is the anti-pattern — it front-loads signup/verify/password/legal-form and shows value only after a multi-day back-office setup. We invert: value (the dashboard) is immediate; paperwork (the LOA) is an opt-in upgrade, never the entry toll. We also never ask the grower to type address/city/zip/phone that is printed on the bill they upload.

## Authentication / login (my judgment, per Jaiyen's steer)
- **Returning customer → straight to their dashboard.** Long-lived session (grower on a phone in a truck should not get logged out).
- **Primary: Continue with Google (OAuth/SSO)** — one tap, nothing to remember.
- **Fallback: magic link** (emailed one-tap sign-in).
- **No passwords at all** — Google SSO + magic link only. Aligns with the "never store credentials" rule; nothing to forget or leak.
- **Operator case:** Jaiyen signs the farmer in with their Google on-site, or sets the farm up under his own account. `[ASSUMPTION]` "manage multiple farms from one login" flagged for the architecture (fits the sales motion).
- **Logged in but no data yet → "connect your data"**, never a dead end.

## Voice & tone
- Plain operator English. No em dashes, no exclamation marks, never salesy. The grower's language: blocks, sets, hours, acres, pumps, ranches. Never kW or "15-minute interval" on the surface.

## Visual system (from project-context "RESOLVED 2026-06-08", plus my calls)
- **Color:** dominant brand green **#2FA84F** on warm paper **#FAF9F4** (never pure white); warm near-black charcoal **#1A1A17** text (never pure black); muted warm gray for secondary text. One brighter green reserved for positive money/savings.
- **Status color (resolving the project-context open item, my call):** keep the green-dominant palette; add ONE restrained warm **clay/terracotta alert tone** for `act` severity and high-$-at-risk map pins (the warm agrarian "clay accent" already named in project-context). `watch` is carried by typography + label, NOT a third color. This delivers the research's instant good/watch/bad without a SaaS rainbow.
- **Typography:** **Helvetica (Helvetica Neue)** across display, body, and data; Arial system fallback. Hierarchy from weight extremes (300/400 vs 700+) and real size jumps, never from mixing families. All numeric/dollar/usage values use **tabular figures**.
- **Layout:** 8px spacing scale; generous margins; hairline 1px low-opacity charcoal borders over filled boxes; soft, low, diffuse shadows only. Mobile-first.
- **Motion:** one orchestrated reveal per view (staggered), easing `cubic-bezier(0.16, 1, 0.3, 1)`, 400–700ms, stagger 60–100ms, no bounce/overshoot. Honor `prefers-reduced-motion` (instant).
- **Aesthetic:** editorial agrarian-luxury. No glassmorphism, liquid glass, or heavy gradients. The attached future-vision mock is a density anti-pattern (12-widget wall, vanity stats) — we lift its skeleton (shell + map + drill-in), not its skin.

## Form factor
- **Mobile-first**, responsive up. The dense multi-column table and three-zone layout are a tablet/desktop **power surface**; mobile core = KPI strip + lens (chart/map/simplified list) + drawer + peeking findings sheet, nav as a bottom tab bar.

## Update 2026-06-15 — "Restrained polish" visual level-up (with Jaiyen)

Triggered by Jaiyen: the dashboard UI reads "extremely old school." Goal is to level it up while keeping the brand. Direction and tooling were chosen by Jaiyen via a structured prompt; this entry is canonical and DESIGN.md is revised from it.

**Direction chosen: RESTRAINED POLISH** (depth + micro-interactions + motion-on-data, Linear/Vercel calm tuned to warm farm paper). Explicitly NOT full Aceternity/marketing-site flash, and not pure editorial-static. Audience unchanged: skeptical, low-software-literacy grower on a phone in a truck.

**Conflict surfaced and resolved (the important one).** The 2026-06-08 spine was anti-effects ("restraint over effects, no gradients, border before shadow, one motion moment, shadows only on the drawer"). The 2026-06-10 CLAUDE.md rewrite retired that restraint and made **Magic UI the primary component/animation vocabulary** ("animated, polished, modern is the goal now, not restraint"). Spine and project law contradicted each other. **Resolution:** the spine now encodes the middle path — keep brand + legibility-first discipline + data-hero principle; retire the blanket no-effects clause; add a real depth scale and a motivated-motion grammar. Glassmorphism / liquid glass / heavy-neon gradients / decorative loops stay banned (the brand and the audience both reject them).

**Tooling decided (all four):**
- **Magic UI** = base component/animation vocabulary, installed via shadcn into `src/components/magicui/` (today's BorderBeam/DotPattern/NumberTicker/AnimatedShinyText were hand-copied into `components/ui/`; formalize via the registry).
- **Aceternity** = shadcn-compatible, pulled for select high-impact moments only, always tinted to the warm palette.
- **Taste Skill** (`design-taste-frontend`, installed at `.agents/skills/`) = anti-slop discipline checklist, NOT an aesthetic. NOTE: it is scoped to landing pages and its defaults *ban our exact warm palette + Inter*; we **consciously override** those because the brand is deliberate and documented. We keep its disciplines: full interaction states, tactile feedback, motion-must-be-motivated, shape/color consistency locks, WCAG AA contrast, copy self-audit. Dials set to its trust-first preset (variance low / motion low / density medium).
- **Impeccable** = audit/clarify/harden disciplines applied as a review pass; the `npx impeccable install` CLI was blocked by the sandbox classifier (runs third-party code), so it is applied manually unless Jaiyen installs it.

**DESIGN.md deltas written this update:**
1. New frontmatter `elevation` scale (`e0-flat`→`e4-modal`, `hover-lift`, `press`, `focus-ring`) and `motion` grammar (5 durations, house easing `cubic-bezier(0.16,1,0.3,1)`, exit easing, 60ms stagger, data spring no-overshoot). Added `gold` (beam-only) color and `rounded.xl`; added `card` + `skeleton` to components.
2. **Brand & Style** reconciled: restraint *with* motivated motion + soft depth; effects must communicate; Magic UI/Aceternity tinted to palette.
3. **Elevation & Depth** rewritten from "border before shadow / drawer-only lift" to the five-step scale; cards now rest at `e1`, lift to `e2`.
4. New **Motion & Animation** section (chrome vs entrance vs data motion; tactile `:active`; reduced-motion non-negotiable; motivated-only rule; BorderBeam as the one decorative exception).
5. **Components** upgraded with concrete specs (card primitive, kpi-card NumberTicker + skeleton, meter-table sticky header + hover accent bar + mobile card-list, cost-chart grow-on-load + hover tooltip, finding-card full state cycle, button tactile + single ShimmerButton CTA, skeleton replaces spinners).
6. **Do's and Don'ts** updated: soft depth on cards; full interaction cycle on every control; WCAG AA checks; don't over-animate tables; reduced-motion everywhere.

**Stale note corrected:** the 2026-06-08 log line resolving the typeface to "Helvetica" is superseded — Inter is the brand-mandated typeface (already reflected in the spine frontmatter and confirmed by project memory).

**Implementation plan:** first vertical slice = Home overview + Energy meter-table, built against this spine, then reviewed through the Taste Skill + Impeccable pre-flight before rollout.
