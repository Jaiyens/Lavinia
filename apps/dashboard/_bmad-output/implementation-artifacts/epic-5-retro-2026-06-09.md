## What went well

1. **Reuse over rewrite.** 5.2's onboarding drove every Epic 1 ingest edge without rewriting one. The Epic 1 trust spine paid off exactly as designed — operator-operable UI was a thin shell over proven importers.
2. **Pure-predicate gates.** `hasRealSource` (5.2) and `showPendingPullBanner` (5.3) were written as pure, tested functions rather than logic buried in Server Actions. Every branch covered; the pending-banner state is proven by test even though the live seed never shows it. "Honest coverage over fake state" held under pressure.
3. **Adapter-collision save.** 5.1's `@auth/prisma-adapter` wanting a delegate named `account` while Terra already had a PG&E `Account` — solved with `AuthAccount` + a Proxy wrapper, documented as intentional variance rather than a silent hack. Reusable pattern.

## What fought back

The expensive bugs were all **structural / isolation**, invisible to the type system:

- **Redirect loop (5.2 — the epic's flagged top risk):** a no-data redirect at the layout level bounced onboarding pages infinitely. Fixed structurally by splitting `(app)/layout.tsx` (auth-only) from `(app)/(dashboard)/layout.tsx` (shell + no-data redirect).
- **Three HIGH cross-tenant IDORs (5.2 x2, 5.3 risk):** connect/confirm pages trusted a URL `farmId` with no ownership check. Patched to `auth()` + `findFirst({ where: { id, userId } })`. Readable data is as sensitive as writable.
- **Public tour rendered live finding buttons (5.3 HIGH):** a read-only surface still rendered Done/Dismiss controls that failed auth for anonymous visitors. Fixed with a `readOnly` prop threaded through `EnergyDashboard -> MeterDrawer -> FindingCard`.
- **Stubs that ignore their input (5.2):** the vision stub ignores bytes, so an empty bill upload silently added a phantom meter; a zero-meter Green Button file silently locked the gate. Both hidden by "it didn't crash."
- **Product-integrity drift (5.2 — AC5):** the "never upload a bill again" LOA pitch had crept onto the PG&E card at the *entry gate*, violating the "LOA is an upgrade after value, never the entry gate" thesis. Review pulled it.

## The one recurring theme

**Green CI is not green security.** Every structural and isolation bug passed lint, tsc, and unit tests, and surfaced only in code review or live `next start` verification. For an app about to hold real growers' utility data, the verification layer that caught these is the layer that matters most. Corollary now treated as pattern: every Server Action opens with `auth()`; every readable page ownership-scopes its query; every public route pins its data source and proves it with a test.

---

## Patterns established (worth reusing)

- **Adapter Proxy wrapper** for third-party model-name collisions (5.1).
- **Split layout** — separate auth gate from data-dependent shell to avoid redirect loops (5.2).
- **Demo-pinning** — `demoFarm(prisma)` + `demoOnly` prop chain, test-proven by `demo.db.test.ts`, to expose demo data on a public route without leaking real data (5.3).
- **Prop-threading feature context** (`demoOnly`, `readOnly`) over per-component conditionals (5.3).
- **Pure tested predicates for gates** (`hasRealSource`, `showPendingPullBanner`).
- **`auth()` re-check opening every Server Action**, since actions are independently reachable.

## Action plan

| # | Item | Why it blocks launch | Where |
|---|------|---------------------|-------|
| C1 | `db:seed` must preserve `isDemo:false` farms | `db:seed` / `db:reset` / auto-seed-on-migrate deletes a real imported account. Smallest fix, highest blast radius — do first. | `prisma/seed.ts` |
| C2 | Multi-tenant ownership filtering | `dashboardFarm`/`currentFarm` populate `Farm.userId` but don't filter by it; a 2nd connected grower lands any user on the most recent farm. Reconcile with the `isDemo` representative fallback. | `src/lib/dashboard` |

### Process / quality

- **A1 — Verification gate for risky changes.** Any auth, routing, or data-isolation change requires a live `next start` round-trip + an integration test before "done." This epic's worst bugs were all invisible to lint/tsc/unit. Standing policy, not per-story luck.
- **A2 — Codify the ownership/auth pattern** into `project-context.md`: every Server Action opens with `auth()`; every readable page ownership-scopes (`findFirst({where:{id,userId}})`); every public route pins its data source and proves isolation with a test.
- **A3 — Stubs validate their input.** A stubbed boundary validates and logs its input even when it discards it (the vision-stub trap).
- **A4 — Repair or retire the legacy e2e.** `e2e/onboarding.spec.ts` still tests the parked pump-timing onboarding; rewrite against the `(app)` onboarding + `/tour`, or retire with the legacy tool.
- **A5 — Land the post-value LOA upsell surface.** Give the (correctly removed) "never upload a bill again" line a calm post-confirm/settings home.

### Secondary debt (track, not launch-blocking)

- Account-level one-cent reconciliation tolerance doesn't scale across dozens of SAs (Batth's real shape) — settle the threshold against the real account.
- NEM extraction at bulk: ~12 graph pages classify as `nem_reconciliation` with empty `monthlyRows`; `generateObject` has no retry/cost budget for the 57-account run.
- `billingSerial -> serialCode` column drop + repoint dormant readers (Epic 5 rewrote the onboarding paths that held them).
- Polymorphic `Button` to fix nested `<Link><Button>` on the tour CTA + 5.2 source picker.
- Orphan pending farms from `identifyFarmAction`; onboarding upload size/count caps; `/tour` empty-state when no demo seed.

## Key takeaways

1. Green CI is not green security — invest in live verification + integration tests for anything touching auth, routing, or data isolation.
2. Reuse beat rewrite: the Epic 1 trust spine made Epic 5 a thin shell. The layered-boundaries discipline paid off.
3. Stubbed boundaries are liabilities until they validate their input.
4. Defer multi-tenant *filtering* deliberately, but it is now the top launch blocker — own it in a dedicated story before a 2nd grower connects.
5. First retro: no prior action items to grade. These action items become the accountability baseline for the next session.
