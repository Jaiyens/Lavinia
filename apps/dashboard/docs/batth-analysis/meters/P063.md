## (b) RATE SCHEDULE decoded in plain English

- **Inventory rate code:** `HAGA2` (PG&E **AG-A2**; the "H" prefix is the internal handle form).
- **Billed/latest tariff (bill print):** **"AGA2 Ag<35 kW High Use"** (PG&E AG-A2).

**What AG-A2 is, plainly:** AG-A2 is PG&E's agricultural time-of-use rate for **small** services
(published eligibility is the **under-35 kW** family, same as AG-A1) but for meters that run
**heavier / steadier ("High Use")**. It has three pieces:
- a daily **customer charge** (here **$0.68895/day**, ≈ $20.96/month),
- **time-of-use energy** $/kWh split into **Peak** and **Off-Peak** windows (the rate peak is
  the 5–8pm window; everything else is off-peak), priced **lower per kWh than AG-A1** (winter
  peak $0.24571 vs AG-A1's $0.33989; winter off-peak $0.21642 vs $0.31060), and
- a **max-demand charge** of **$21.43/kW** on the highest demand in the cycle — notably **higher**
  than AG-A1's $12.33/kW.

So AG-A2 is the classic trade the "High Use" name implies: **cheaper energy, pricier demand**.
It wins over AG-A1 once a meter runs enough kWh that the energy savings outweigh the bigger
demand charge.

**Why this meter is on AG-A2:** a 1,000 GPM well, when it *is* irrigating, moves a lot of kWh —
exactly the "High Use" profile AG-A2 is built for. The inventory code (AG-A2) and the billed
tariff (AG-A2) **agree** — no mis-rating on file. The one caveat is that this single cycle is
**idle**, so we cannot see the in-season kWh that would prove AG-A2 beats AG-A1 here; that
requires summer data (see finding 1).

- **Engine:** **`rateLever()` → `backTestMeter()` in `src/lib/energy/rate-lever.ts`**

  `src/lib/recommendations/run-rate-lever.ts`.
- **Arithmetic (traced against this meter's data):**
  - The lever maps `AGA2` → card plan **AG-A2** (small real-tier, large card row), then
    recomputes the cycle from the card: customer = 30 days × $0.68895 = **$20.67**, energy = $0,
    demand = $0 → **recomputed $20.67 vs printed $20.67 = 0.00% deviation**, comfortably inside
    the **5% `BACK_TEST_BAND_PCT`**. The gate **passes**.

    (AG-A2 is a *current* non-legacy plan and total tested kWh = 0), so it returns
    `kind: "none", reason: "no_usage_basis"`. An idle winter meter's customer-charge delta is
    **not** a defensible reason to switch between current usage-tiered schedules (summer flips
    it). No dollar is quoted.
  - For completeness: pricing this same idle cycle on the small-tier candidates gives AG-A1 =

- **Data we HAVE:** one reconciled, **idle** winter cycle (0 kWh), the rate codes (agree), the
