## (b) RATE SCHEDULE decoded in plain English

- **Inventory rate code:** `HAGB` (the "H" prefix is PG&E's internal handle; the schedule is **AG-B**). The bill prints / billing engine's `latestTariff` is **"AGB Ag35+ kW Med Use"**.
- **What AG-B is, in operator English:** AG-B is PG&E's **medium agricultural** time-of-use schedule for **35 kW and up** service ("Ag35+ kW"). Like AG-C it is a demand-charge-carrying ag rate: a small **daily customer charge**, per-kWh **energy priced by time-of-use** (peak vs off-peak), and a monthly **maximum-demand charge in $/kW** on the highest 15-minute kW the meter pulls in the cycle. AG-B's demand rate is *lower* than AG-C's ($13.95/kW vs $24.95/kW on the card) and its energy is *higher* — AG-B suits meters that pull a modest peak but want a cheaper demand component; AG-C suits big steady pulls that want cheap energy and accept a steep demand charge. (That AG-B vs AG-C trade-off is exactly what the rate-optimization lever in section (e) is built to test once interval data exists.)
- **Customer charge:** the AG-B card models **$0.91565 per day** (~$27.85/month). On this bill the two dated customer-charge sub-periods sum to **$27.47** for the 30-day cycle (see section c), consistent with that daily rate across the 2026-03-01 rate change.
- **Max-demand $/kW:** the AG-B card max-demand is **$13.95/kW** (bill-sourced, post-03/01/2026 change; AG-B is not tiered by size, so AG-B1 and AG-B2 carry the same prints). On this cycle the printed structured demand charge was **$0.17 on 0.012 kW** — an implied ~$14/kW, in line with the card. There is no separate summer peak-period demand adder on AG-B (unlike AG-C).
- **TOU windows:** the **AG RATE peak is 5–8pm year-round daily** (this is the energy/demand pricing clock, and is distinct from the 4–9pm PDP/DR *event* window). This bill carries only **Peak** and **Off-Peak** energy buckets (no partial-peak rows). Printed AG-B winter energy on THIS bill was **$0.34015/kWh peak** and **$0.31089/kWh off-peak** (the card's reference AG-B winter figures are $0.33597 peak / $0.30671 off-peak — the small gap is the usual print-vs-card drift the back-test band absorbs).
- **WHY this meter is on AG-B:** it is a 35 kW+ eligible agricultural pump on the master account that pulls only a modest demand. AG-B (cheaper demand, pricier energy) is a reasonable fit for a pump that runs intermittently and rarely sets a big peak. Whether AG-B is the *cheapest* eligible current schedule for its real annual usage is precisely the open question the rate lever answers — but only with interval data (see e). On the bill as printed, nothing screams mis-rated.

## (e) EVERY applicable savings finding FOR THIS METER (full arithmetic + engine)

All dollar figures below come from **deterministic pure functions in `src/lib/energy`** (no AI). The only AI in the pipeline is bill-PDF vision extraction in `src/lib/extract`. The orchestrator is `src/lib/recommendations/run.ts::runEngines`.

1. **Rate optimization (`rate-lever.ts::rateLever` / `rate-compare.ts::rateOptimization`, tool `rate-optimization`)** — **NO FINDING (gated out: no intervals).**
   - `runEngines` only runs the rate lever when `pump.rateSchedule && isAg(schedule) && intervals.length > 0 && pump.solarKw === null`. P058 **is** AG (`HAGB`/AG-B) and **is** non-solar (`solarKw === null`), so it clears those two gates — **but it has no 15-minute interval history** (`intervals.length === 0`), so the lever never runs. This is by design: per ground truth, an AG-B↔AG-C "savings" computed without intervals is a **sign-ambiguous artifact** and is suppressed.
   - Even if the lever were forced to run on the bill summary alone: P058 has only **one** reconciled cycle with **5.327 kWh** of tested energy. `rateLever` maps `AGB → AG-B2` (large tier, `realTier: "large"`), back-tests the single cycle against the card, and — for a non-legacy current plan — applies the `totalKwhTested <= 0` guard. 5.327 kWh is barely above zero; even if it passed the back-test band, the candidate set is `{AG-A2, AG-C2}` (large-tier current ag schedules), and over a near-zero-usage winter cycle the only delta is the customer-charge difference, which the lever explicitly refuses to quote (`"a current-schedule swap must rest on real usage"`). It would return `reason: "no_savings"` or be dominated by the `savingsFloorCents = max($1, sumAbsError)` floor. **Result: no defensible finding.**
   - **Determinism:** pure, deterministic; no AI. **Data we HAVE:** AG-B rate label, one reconciled cycle, peak/off-peak kWh, demand $0.17. **Data we NEED:** **15-minute interval kWh** (none on file) and ideally a **summer cycle** (a winter-only meter must never be downgraded to a small-tier schedule — winter demand understates summer). Without intervals, no honest rate-switch dollar.

2. **Demand-charge exposure (`retrospective.ts::retrospective`, tool `demand-charge`)** — **NO FINDING (gated out: no intervals).**
   - Gated on `intervals.length > 0 && pump.solarKw === null`. P058 is non-solar but has **no intervals**, so the lever never runs. There is nothing to attribute a peak to.
   - The billed demand is microscopic anyway: **$0.17 on a 0.012 kW** max demand. There is no avoidable-spike story here — the meter is essentially idle. **Determinism:** pure. **HAVE:** the structured demand line ($0.17, 0.012 kW). **NEED:** interval history to identify which day/hour set the (trivial) peak. **No finding.**

3. **Bill audit (`bill-audit.ts::billAudit`, tool `bill-audit`)** — **NO FINDING (single cycle, no comparators).**
   - This is the one engine with no interval gate — it runs on every pump that has bills and compares each cycle's total to the **median of the meter's OTHER same-season cycles**. P058 carries a peak (`peakKw = 0.012 > 0`), so it would take the **with-peak path**, which requires `minComparators = 3` same-season peer cycles with a usable peak. **P058 has exactly one cycle** (`cycleCount: 1`) — zero peers — so `peers.length (0) < 3` and the engine `continue`s immediately. **No flag.**
   - **Determinism:** pure median math, no AI. **HAVE:** one printed total ($29.31) with its peak. **NEED:** at least 3 more same-season billed cycles to build a stable median. **No finding.**

4. **Solar / NEM checks (`solar-nem.ts::solarNemChecks` and canonical `run-solar-insight.ts`, tool `solar`)** — **NOT APPLICABLE.**
   - Both the legacy demo-only `solarNemChecks` (gated `isDemo && pump.solarKw !== null`) and the canonical `runSolarInsight` operate only on solar-paired meters. **P058 has `solarKw === null` and `solarFlag === false`** ("not using"), so neither engine touches it. **No finding, by construction.**

**Net for P058:** **no >$500/yr deterministic finding.** It is a small, near-idle, non-solar SWANSON pump on AG-B with only a single $29.31 winter cycle on file.
